## Executive conclusion

The repository has a strong dotfile-management core. Per-file symlinking,
collision-safe backups, physical-path ownership checks, manifest-backed
uninstall, and backup restoration are notably better than the average personal
dotfiles repository.

The main weakness is that four different concerns are coupled together:

1. linking user configuration;
2. installing packages;
3. provisioning macOS system preferences; and
4. applying security hardening.

That coupling makes the default macOS path too invasive, causes routine updates
to repeat privileged setup, and makes the meaning of profiles inconsistent.
The management CLI, hooks, and health checks then use different notions of the
active profile and of which paths the repository owns.

The highest-value direction is to make symlink reconciliation the safe default,
move system provisioning behind explicit operations, and drive every lifecycle
command from one declarative source-to-destination manifest.

## Strengths

### S-01 — Symlink creation and backup handling

`install/symlinks.sh` creates links per file instead of replacing whole config
directories. Existing files are moved to unique timestamped backup names, with
collision suffixes when several backups occur in the same second.

### S-02 — Safe uninstall ownership model

Uninstall removes only links whose targets are under the physical repository
root. Foreign symlinks and real files survive, and the newest prior backup is
restored without overwriting an existing destination.

### S-03 — Useful regression coverage

The suite contains 47 bootstrap/profile tests and 73 uninstall tests. It covers
profile expansion, dry runs, CLI failures, collision handling, physical path
resolution, foreign-link preservation, manifest behavior, scoped uninstall,
and backup restoration.

### S-04 — Cross-platform CI foundation

CI runs behavioral tests on both Ubuntu and macOS, checks the principal shell
scripts with ShellCheck, parses Zsh configuration, and parses Neovim Lua.

### S-05 — Good Git defaults

`zdiff3`, histogram diffs, rerere, automatic upstream creation, fetch pruning,
autostash, and a default `main` branch are strong defaults.

### S-06 — Clear configuration structure

Zsh and Neovim are modular rather than monolithic. The tmux configuration is
deliberately small and validated successfully with `tmux-256color`, truecolor,
focus events, and mouse support.

### S-07 — Secret hygiene

No obvious tracked credentials or private keys were found. Machine identity and
signing configuration are kept in ignored `gitconfig.local` state.

### S-08 — Current FPATH worktree change is sound

The uncommitted addition of `$HOMEBREW_PREFIX/share/zsh/functions` uses a stable
Homebrew symlink instead of a version-pinned Cellar path. This should prevent
autoload failures in existing shells after a Homebrew Zsh upgrade.

## Findings: installation and system provisioning

### BOOT-01 — Default macOS bootstrap applies system preferences

- Severity/status: **Critical / Confirmed**
- Evidence: `bootstrap.sh:413-424`, `install/install-macos.sh:57-74`

A normal bootstrap calls the OS installer before linking configuration. On
macOS, the installer always requests sudo and executes
`config/macos/defaults.zsh`; only the separate hardening script has an opt-in
prompt. The README quick start presents plain `./bootstrap.sh` as the normal
installation path and describes installation as safe because user files are
backed up.

The defaults script changes NVRAM, power management, application quarantine,
system services, login-window behavior, and application preferences. These are
system-provisioning actions, not ordinary dotfile linking, and they have no
transaction or rollback boundary.

Recommendation: make link-only installation the default. Require separate,
explicit operations such as `--install-packages`, `--apply-macos-defaults`, and
`--harden`.

### BOOT-02 — Touch ID setup overwrites existing PAM configuration

- Severity/status: **Critical / Confirmed**
- Evidence: `config/macos/defaults.zsh:266-271`

Every defaults run copies `/etc/pam.d/sudo_local.template` over
`/etc/pam.d/sudo_local`, then uncomments every `auth` line. This destroys
existing additions such as `pam_reattach`, has no backup, and is not guarded for
macOS releases where the template may not exist.

Recommendation: back up the existing file, gate by OS version and template
availability, and insert only the required Touch ID line if it is not already
present.

### BOOT-03 — macOS defaults are not fail-fast or reliably idempotent

- Severity/status: **High / Confirmed**
- Evidence: `config/macos/defaults.zsh`

The script has no strict error mode. Unsupported or failing operations can be
followed by a successful final message. Current examples include legacy
Notification Center and `com.apple.rcd` unloads, application-domain writes that
may no longer be effective, and `PlistBuddy Set` operations whose nested keys do
not exist on a fresh machine.

Recommendation: split settings into small version-gated functions, fail or
report accurately per setting, use `Set || Add` where appropriate, and record
the previous value of each changed setting.

### HARD-01 — Hardening backup cannot be created by the invoking user

- Severity/status: **Critical / Confirmed**
- Evidence: `config/macos/harden.zsh:37-43`

`backup_settings` chooses `/var/root/security_backup_<timestamp>` and then runs
plain `mkdir` and shell redirections without sudo. The script is launched as the
interactive user, so the backup normally fails before privileged changes begin.

Recommendation: write to a user-owned, mode-`0700` backup directory or perform
the complete backup through one audited privileged helper.

### HARD-02 — Hardening failures can still produce a success result

- Severity/status: **High / Confirmed**
- Evidence: `config/macos/harden.zsh:61-66`, `130-149`

The script lacks `set -euo pipefail`. `execute_command` logs failures, but
callers continue through loops and later groups. The final successful command
can make the outer block succeed, after which the script logs that hardening
completed successfully.

The backup covers only user defaults and `pmset`, not password policy, disabled
services, firewall state, or other changes, and there is no restore command.

Recommendation: propagate every failed command, model the desired state and
prior state explicitly, and test both failure and restoration paths.

### UPDATE-01 — Routine updates repeat installation and system provisioning

- Severity/status: **High / Confirmed**
- Evidence: `Makefile:77-82`

`make update` performs `git pull --ff-only` and then runs a normal bootstrap.
It does not use the stored profile, defaults to `minimal`, repeats package
installation, and on macOS repeats the defaults script.

Recommendation: make update pull and reconcile the stored profile in sync mode.
Package and system updates should be separately named targets.

### UBUNTU-01 — Repository Neovim cannot run on supported distro packages

- Severity/status: **Critical / Confirmed**
- Evidence: `install/install-ubuntu.sh:21-44`,
  `config/nvim/lua/kostevski/plugins/lsp/lspconfig.lua:133-165`,
  `config/nvim/lua/kostevski/config/keymaps.lua:46-50`

Ubuntu setup installs the distribution `neovim` package without a version
check. The config uses `vim.lsp.config`, `vim.lsp.enable`, and
`vim.diagnostic.jump`, requiring Neovim 0.11 or newer. Ubuntu Jammy provides
0.6.1, Debian Bookworm provides 0.7.2, and Debian stable source currently
provides 0.10.4.

Recommendation: install a pinned official 0.11+ or 0.12 release artifact and
enforce the version floor during bootstrap and at Neovim startup.

### UBUNTU-02 — Optional network additions can abort the entire setup

- Severity/status: **Medium / Confirmed**
- Evidence: `install/install-ubuntu.sh:46-76`

The Ubuntu installer runs with `set -euo pipefail`. A transient failure while
adding the eza repository, refreshing apt metadata, or installing `thefuck`
through pipx aborts the whole installation after earlier packages have already
changed the system. The configured eza repository URL also uses HTTP rather than
HTTPS, although the package repository is key-signed.

Recommendation: distinguish required and optional packages, use retries for
network operations, prefer HTTPS, and summarize optional failures at the end.

### PACKAGE-01 — Profiles do not consistently control installed packages

- Severity/status: **Medium / Confirmed**
- Evidence: `install/homebrew.sh:81-96`, `config/homebrew/Brewfile-min:1-25`,
  `install/install-ubuntu.sh:21-44`

On macOS, `minimal` and `standard` use the same Brewfile, and that minimal file
installs Neovim, Node, and Python despite the profile description listing only
Git, Zsh, and tmux. On Ubuntu, every profile receives the same package list.

Recommendation: either rename profiles as *link profiles* and document package
installation separately, or define package tiers alongside config tiers.

### HOMEBREW-01 — Persistent Homebrew environment setup is split and ordered poorly

- Severity/status: **Medium / Confirmed**
- Evidence: `install/homebrew.sh:38-55`, `config/homebrew/brew.env`,
  `bootstrap.sh:420-424`

The installer creates `~/.config/homebrew/config`, which current Homebrew does
not document as an environment file. The tracked `brew.env` is at the correct
XDG location once the full profile links it, but linking happens only after the
first `brew bundle` run. Minimal and standard profiles do not link the Homebrew
config directory at all.

Recommendation: use only `$XDG_CONFIG_HOME/homebrew/brew.env`, create or link it
before invoking Homebrew, and make its profile scope explicit.

### MAP-01 — Top-level configuration files are silently un-installable

- Severity/status: **High / Confirmed**
- Evidence: `install/profiles.sh:12-16`, `27-58`, `bootstrap.sh:246-259`

The full profile lists `clang-format`, but `link_configs` only processes
directories. `config/clang-format` is therefore skipped silently; the bootstrap
test explicitly exempts it from the directory-existence invariant. The
top-level `config/.curlrc` likewise has no mapping to `~/.curlrc`.

An isolated full-profile sync confirmed that neither clang-format nor curl
configuration was installed.

Recommendation: replace directory inference with a mapping containing source,
destination, profile, platform, and path type. Reject invalid profile entries.

### DEST-01 — Custom destination flags are only partially honored

- Severity/status: **Medium / Confirmed by static trace**
- Evidence: `bootstrap.sh:28-35`, `bin/dotfiles:41-44`,
  `install/symlinks.sh:125-178`

`--config-dest` and `--bin-dest` affect linking, but the manifest, stored
profile, status, cleanup, uninstall, and some directory creation continue to
derive paths directly from `$HOME`.

Recommendation: thread an installation context through every operation or
remove the flags until full alternate-destination support exists.

### PROFILE-04 — Interactive and custom profile code is dead surface

- Severity/status: **Low / Confirmed**
- Evidence: `install/profiles.sh:183-367`, `369-382`

The interactive selectors, current-profile detector, custom component array,
and related presentation code have no caller from bootstrap or the management
CLI. `custom` still validates, but a non-interactive custom profile has no
selected components and falls back to minimal behavior.

Recommendation: wire the selector into an explicit interactive command or
delete the subsystem and reject `custom`.

### SCAFFOLD-01 — Empty configuration trees add false surface

- Severity/status: **Low / Confirmed**
- Evidence: `config/defaults/`, `config/security/`, `config/ubuntu/`

These directories contain no tracked configuration but participate in repo
discovery and make it less clear which components are supported.

Recommendation: delete them until a real configuration needs the namespace.

## Findings: profiles, sync, status, and ownership

### PROFILE-01 — Switching the stored profile does not apply it

- Severity/status: **High / Reproduced**
- Evidence: `bin/dotfiles:52-65`, `67-119`, `274-305`

`dotfiles profile standard` stores `standard` and tells the user to run
`dotfiles sync`. Default sync ignores the stored value and invokes bootstrap
with `--synced`, which only reconciles configurations already linked.

Reproduction in an isolated home:

1. install `minimal`;
2. run `dotfiles profile standard`;
3. run `dotfiles sync`;
4. observe that Neovim is still absent.

Recommendation: make default sync apply the stored profile. Preserve
`--synced` as an explicitly named installed-links-only mode.

### PROFILE-02 — Hooks and watch mode are profile-blind

- Severity/status: **Medium / Confirmed**
- Evidence: `.githooks/post-checkout:12-21`, `.githooks/post-merge:9-15`,
  `bin/dotfiles:251-271`

All three call bare `bootstrap.sh --sync`, whose default profile is minimal.
Edits to existing linked files are immediately visible through their symlinks,
but new standard/full-profile files are not linked.

Recommendation: route every automatic sync through the same stored-profile
resolver as the CLI.

### PROFILE-03 — Hook change detection omits lifecycle inputs

- Severity/status: **Low / Confirmed**
- Evidence: `.githooks/post-checkout:12-14`, `.githooks/post-merge:9-10`

Hooks trigger only for paths under `config/`. Changes to `install/profiles.sh`,
the future mapping/manifest, `bin/`, or bootstrap behavior can require a sync
without triggering one.

Recommendation: maintain a short explicit list of lifecycle inputs or compute
whether the expected manifest changed.

### CLEAN-01 — Every sync can remove unrelated broken symlinks

- Severity/status: **High / Confirmed**
- Evidence: `bootstrap.sh:385-398`, `install/symlinks.sh:125-178`

Sync begins by finding every broken symlink under `~/.config` and
`~/.local/bin` and deleting it. The links do not need to point into this
repository or appear in its manifest. Tests currently encode removal of
arbitrary broken links as intended behavior.

Recommendation: clean only manifest-owned destinations. Offer an explicit
global broken-link cleanup command if that behavior remains desirable.

### STATUS-01 — Status is not profile-aware and cannot fail automation

- Severity/status: **High / Reproduced**
- Evidence: `bin/dotfiles:147-249`

Status enumerates every top-level config directory, including `config/macos`,
which no profile links as ordinary XDG configuration. In an isolated full
installation it reported 156 correct links, zero broken links, and two missing
macOS files. It still exited with status zero.

Recommendation: derive expected paths from the installed/stored profile or
manifest and return nonzero when any expected destination is missing or wrong.

### CLI-01 — Unknown management commands exit successfully

- Severity/status: **Medium / Reproduced**
- Evidence: `bin/dotfiles:428-475`, `478-521`

The unknown-command branch prints an error and then calls `usage`, which always
exits zero. Scripts and users therefore cannot distinguish invalid invocation
from successful help output.

Recommendation: make `usage` accept an exit code or return to the caller.

### VALIDATE-01 — Make validation reports failure without failing

- Severity/status: **Medium / Confirmed**
- Evidence: `Makefile:88-108`

`make validate` prints broken links but exits successfully. Both validation and
clean also scan all links under the destination trees instead of repository
ownership. `make clean` duplicates the lifecycle implementation with bare
`rm`.

Recommendation: delegate both targets to `bin/dotfiles`, preserve ownership
boundaries, and use meaningful exit status.

### HOOK-01 — Hook activation is not documented in the normal setup flow

- Severity/status: **Low / Drift**
- Evidence: `.githooks/setup.sh`, `README.md`

Hooks require `git config core.hooksPath .githooks`, but the README does not
tell a fresh user to run the setup script. The current checkout happens to have
the local setting enabled; a clone will not.

Recommendation: document the optional hook setup and its profile semantics.

## Findings: Git

### GIT-01 — Global ignores hide legitimate project content

- Severity/status: **High / Confirmed**
- Evidence: `config/git/ignore`

Git treats `$XDG_CONFIG_HOME/git/ignore` as its default per-user excludes file.
This file globally ignores `*.sql`, `*.sqlite`, `*.jar`, archives, `.vscode/`,
and other paths that projects commonly commit. New migrations, fixtures,
vendored artifacts, or shared editor settings can disappear from `git status`
and ordinary `git add`.

Recommendation: restrict the global file to OS metadata, editor backups, and
genuinely universal private state. Put language and artifact rules in each
project.

### GIT-02 — Windows metadata pattern contains a typo

- Severity/status: **Low / Confirmed**
- Evidence: `config/git/ignore:35`

`Thumbs.db.DS_Store` appears to be two concatenated patterns and matches neither
normal Windows nor macOS metadata as intended.

### GIT-03 — Git LFS is required globally but not installed by the setup

- Severity/status: **Medium / Confirmed**
- Evidence: `config/git/config:166-170`, both Brewfiles,
  `install/install-ubuntu.sh`

The global filter declares `required = true`, but package definitions do not
guarantee `git-lfs`. Repositories containing LFS pointers can fail to check out
or commit on a freshly provisioned machine.

Recommendation: install Git LFS on every supported platform and run
`git lfs install`, or remove the global filter until needed.

### GIT-04 — Delete-merged alias can remove protected local branches

- Severity/status: **Medium / Confirmed**
- Evidence: `config/git/config:58-60`

`git dm` deletes every merged branch except the current one. When invoked from a
feature branch it can delete local `main`, `master`, or `develop`. It also calls
`xargs` on empty input and relies on pipeline behavior that differs by platform.

Recommendation: exclude protected names explicitly and use a shell loop that
handles an empty set.

### GIT-05 — Machine-local Git onboarding is easy to miss

- Severity/status: **Low / Confirmed**
- Evidence: `config/git/config:161-164`, `install/lib.sh:125-145`

The config includes relative `gitconfig.local`, while `*.local` files are
intentionally not linked. A fresh installation links the example but does not
create the included file. The inline comment is the only onboarding path.

Recommendation: document the exact destination
`~/.config/git/gitconfig.local` and provide a non-secret initializer command.

## Findings: Neovim

### NVIM-01 — Plugin revisions are not reproducible

- Severity/status: **High / Confirmed**
- Evidence: no tracked `config/nvim/lazy-lock.json`;
  `config/nvim/lua/kostevski/config/lazy.lua:6-41`

lazy.nvim is cloned from the moving `stable` branch, plugin versions default to
latest commits, and no lockfile records resolved revisions. Fresh machines and
restores can receive different plugin graphs.

Recommendation: commit `lazy-lock.json`, use `:Lazy restore` in bootstrap or
recovery documentation, and define an intentional update workflow.

### NVIM-02 — CI parses Lua but never loads Neovim

- Severity/status: **High / Confirmed**
- Evidence: `.github/workflows/ci.yml:23-31`

`luac5.3 -p` detects syntax only. It does not use Neovim's LuaJIT/runtime,
resolve `require` calls, construct the lazy.nvim graph, start LSP configuration,
or detect invalid plugin modules.

Recommendation: add a pinned Neovim headless smoke test that loads the config
with network/plugin installation disabled or with a restored lockfile.

### NVIM-03 — Generated neotest adapter module names are wrong

- Severity/status: **High / Latent**
- Evidence: `config/nvim/lua/kostevski/utils/lang.lua:283-297`, Python and
  JavaScript language definitions

The adapter helper extracts `neotest-python` or `neotest-jest`, strips the
`neotest-` prefix, and attempts `require("python")` or `require("jest")`.
Official adapter modules are `require("neotest-python")`,
`require("neotest-jest")`, and similar.

The defect is dormant because Python and JavaScript are not currently enabled.

Recommendation: keep the plugin basename intact or allow each language to
declare the Lua module explicitly; cover every configured adapter with a
headless load test.

### NVIM-04 — Docker language module is copy-paste corrupted

- Severity/status: **Medium / Latent**
- Evidence: `config/nvim/lua/kostevski/plugins/lang/docker.lua`

The Docker module declares the name and filetype `Caddyfile`, uses `gopls`, and
selects Go formatters while also declaring Docker root markers and hadolint.

Recommendation: replace it with Dockerfile/Compose-specific filetypes, LSP,
formatters, linting, and parser definitions before enabling it.

### NVIM-05 — Native LSP configs contain lspconfig-only hooks

- Severity/status: **Medium / Latent**
- Evidence: `config/nvim/lsp/jsonls.lua`, `yamlls.lua`, `pyright.lua`

These configs rely on fields such as `on_new_config` and legacy debounce flags
from nvim-lspconfig. The native `vim.lsp.config`/`vim.lsp.enable` path does not
invoke those lspconfig-specific hooks, so SchemaStore enrichment and related
behavior will not occur when the corresponding languages are enabled.

Recommendation: migrate behavior to native callbacks/settings and test the
effective client configuration.

### NVIM-06 — Enabled language support is much narrower than advertised

- Severity/status: **Medium / Drift**
- Evidence: `config/nvim/lua/kostevski/config/languages.lua:12-14`,
  `README.md:5-12`, `144-149`

Only Lua, Terraform, and C++ are enabled. The README advertises full Go, Python,
Rust, and broader multi-language LSP support.

Recommendation: document enabled-by-default versus available modules, or enable
and runtime-test the advertised set.

### NVIM-07 — Python provider instructions and configured path disagree

- Severity/status: **Medium / Confirmed**
- Evidence: `config/nvim/lua/kostevski/config/options.lua:4-7`,
  `README.md:186-198`

Neovim points to `~/.local/share/uv/venv/py3nvim/bin/python`, while the README
creates `~/.local/share/nvim-venv`. Following the documented steps leaves the
configured provider missing.

Recommendation: choose one path, create it through an idempotent command, and
verify with `:checkhealth provider`.

### NVIM-08 — AI integration documentation is stale

- Severity/status: **Low / Drift**
- Evidence: `README.md:144-149`,
  `config/nvim/lua/kostevski/plugins/ai.lua`

The README lists Copilot and Aider, but the active AI plugin file contains only
`greggh/claude-code.nvim`.

### NVIM-09 — Declared StyLua style is not applied to the tree

- Severity/status: **Medium / Confirmed**
- Evidence: `config/nvim/stylua.toml`; local `stylua --check`

The config declares two-space indentation. StyLua reports code-style problems
in 57 Neovim Lua files, and CI has no format gate.

Recommendation: perform one mechanical format commit, then add
`stylua --check config/nvim` to CI.

### NVIM-10 — Minimum Neovim version is not documented or enforced

- Severity/status: **High / Confirmed**
- Evidence: modern APIs in LSP and keymap modules; Ubuntu installer; README

The local machine runs Neovim 0.12.4, but portability depends on an implicit
0.11 floor. Some comments still imply 0.10 compatibility.

Recommendation: state the minimum in README and fail early with a clear message
when it is not met.

### NVIM-11 — Tool and plugin supply is broadly floating

- Severity/status: **Medium / Risk**
- Evidence: lazy.nvim defaults, Mason-managed tools, language modules

Plugins resolve moving Git commits and Mason installs current tool versions.
This increases update convenience but makes recovery and cross-machine parity
unpredictable.

Recommendation: lock plugins, document Mason update policy, and consider a
small version manifest for high-impact formatters, debuggers, and language
servers.

## Findings: Zsh

### ZSH-01 — Fresh startup writes into a missing cache directory

- Severity/status: **High / Reproduced**
- Evidence: `config/zsh/rc.d/30-completions.zsh:27-44`

The completion setup writes `.compdump_check` under `$XDG_CACHE_HOME/zsh`
without creating the directory. A fresh isolated Linux-style home produced a
`no such file or directory` error during shell startup.

Recommendation: create the cache directory with mode `0700` before any compdump
or cache operation.

### ZSH-02 — Runtime directory fallback is shared across Linux users

- Severity/status: **High / Confirmed**
- Evidence: `config/zsh/zshenv:9-11`

When systemd has not supplied `XDG_RUNTIME_DIR` and `TMPDIR` is unset, the
fallback is `/tmp/run`. The first user may create it mode `0700`; later users
then export a directory they cannot access. The path is also too predictable
for sensitive runtime sockets.

Recommendation: use an existing system runtime directory or a UID-specific,
ownership-checked directory such as `${TMPDIR:-/tmp}/run-$UID`.

### ZSH-03 — Repository path conflicts with the documented clone path

- Severity/status: **Medium / Confirmed**
- Evidence: `config/zsh/rc.d/20-exports.zsh:1-4`, `README.md:28-40`

`DOTDIR` is hardcoded to
`~/dev/repos/github.com/a-kostevski/dotfiles`, while quick start clones to
`~/.dotfiles`. Navigation and config-editing aliases therefore fail for users
following the documentation.

Recommendation: derive the repository root from the linked `dotfiles` command
or manifest, or document one canonical location.

### ZSH-04 — Locale is forced without verifying availability

- Severity/status: **Medium / Risk**
- Evidence: `config/zsh/zshenv:87-90`

Both `LANG` and `LC_ALL` are forced to `en_GB.UTF-8`, but Ubuntu setup does not
generate that locale. Minimal servers can emit locale warnings or cause tools
to fall back unexpectedly.

Recommendation: set `LANG` only when available and avoid globally forcing
`LC_ALL` outside commands that require deterministic locale behavior.

### ZSH-05 — Several aliases are broken or unsafe as written

- Severity/status: **Medium / Confirmed**
- Evidence: `config/zsh/rc.d/60-aliases.zsh`

- `clconfig` changes to an unquoted path containing spaces.
- `macos_config` omits the `.zsh` suffixes and sources privileged setup scripts
  into the interactive shell.
- `geoip` and `publicip4` use implicit HTTP or explicit HTTP.

Recommendation: use functions for multiword paths and consequential commands,
remove the direct macOS provisioning alias, and use HTTPS endpoints.

### ZSH-06 — fzf initialization invokes the generator twice

- Severity/status: **Low / Confirmed**
- Evidence: `config/zsh/rc.d/71-plugins.zsh:13-20`

`fzf --zsh` is executed once as a feature check and again to generate code.

Recommendation: capture the output once and source it only when generation
succeeds.

### ZSH-07 — Completion cache invalidation is time-based rather than content-based

- Severity/status: **Medium / Confirmed**
- Evidence: `config/zsh/rc.d/30-completions.zsh:29-44`

The compdump is regenerated once per calendar day. A plugin or package that
adds a completion later that day is not registered until the next day. The
daily full `compinit` can also prompt for insecure Homebrew directories because
the intended policy is not explicit.

Recommendation: invalidate when an `fpath` directory is newer than the dump and
choose an explicit `compaudit` policy.

### ZSH-08 — Zsh plugins are cloned from unpinned HEAD and immediately sourced

- Severity/status: **Medium / Risk**
- Evidence: `config/zsh/rc.d/70-zsh-unplugged.zsh:7-46`

First shell startup clones shallow GitHub HEADs and sources them. There is no
revision manifest or integrity boundary, and updates pull moving branches.

Recommendation: pin known revisions or maintain a lock manifest and update it
through an explicit command.

### ZSH-09 — Smaller startup correctness and maintenance issues remain

- Severity/status: **Low / Confirmed by static review**

- `_plug_compile` is defined but unused.
- `fzf`, 1Password, and completion initialization mix generation and startup
  policy across several files.
- `command_exists` caching can become stale after installing a command in the
  current shell.
- PATH/FPATH duplicate checks use regular-expression matching even though Zsh
  unique arrays can handle exact deduplication.
- Several files use top-level `local`, which is legal in Zsh but obscures the
  resulting scope and complicates sourcing behavior.

These are lower priority than the fresh-start and runtime-directory defects.

## Findings: Kitty, Homebrew, and helper scripts

### KITTY-01 — Editor path is Apple-Silicon-specific

- Severity/status: **Medium / Confirmed**
- Evidence: `config/kitty/kitty.conf:1-3`

Kitty hardcodes `/opt/homebrew/bin/nvim`. It fails on Intel macOS, non-Homebrew
Neovim installations, and Linux machines using the `all` profile.

Recommendation: use `nvim` from `PATH` or generate a machine-local override.

### KITTY-02 — Remote-control socket should use the runtime directory

- Severity/status: **Low / Risk**
- Evidence: `config/kitty/kitty.conf:45-48`

Remote control is enabled and the socket is placed directly under `/tmp`.
`{kitty_pid}` prevents collisions, but an owned `XDG_RUNTIME_DIR` is a clearer
security and lifecycle boundary.

### BIN-01 — `confirm` loops forever on EOF

- Severity/status: **Medium / Confirmed by control flow**
- Evidence: `bin/confirm:3-12`

`read` failure is not handled. With closed stdin, the loop repeatedly prints the
prompt and error message.

Recommendation: treat EOF as decline or return a distinct failure status.

### BIN-02 — `nshift for` validates duration after arithmetic evaluation

- Severity/status: **Medium / Confirmed by control flow**
- Evidence: `bin/nshift:88-118`

The input is evaluated arithmetically before its numeric format is checked. The
validation then depends on `bc`, which is not checked or installed explicitly.
The predictable PID file under `/tmp` can also refer to an unrelated reused PID
if stale or replaced.

Recommendation: validate first, avoid `bc` for the supported grammar, and keep
PID state in an owned runtime directory with process identity verification.

### BIN-03 — Duplicate clock toggle has weaker state handling

- Severity/status: **Low / Confirmed**
- Evidence: `bin/osx-clock-toggle`, `bin/cantsleep`

`osx-clock-toggle` uses `grep -q '1'`, which can match values such as `10`, and
writes the preference as an integer while `cantsleep` uses booleans for the same
key.

Recommendation: delete the duplicate or make it a compatibility wrapper around
`cantsleep`.

### BIN-04 — Smaller executable scripts are outside the normal ShellCheck gate

- Severity/status: **Low / Confirmed**
- Evidence: `Makefile:188-193`, `.github/workflows/ci.yml:14-21`

The gate covers `bin/dotfiles` but not the remaining shell executables. Direct
review found declaration-and-assignment warnings in `bin/cantsleep`; more
importantly, future defects in these scripts will not be linted in CI.

Recommendation: discover eligible shell scripts by shebang and lint them with
documented exclusions for Zsh-only files.

## Recommended remediation order

### Phase 0 — Make installation safe

1. Make link-only installation the default.
2. Put packages, macOS defaults, and hardening behind distinct explicit flags or
   commands.
3. Fix PAM handling and hardening backup/error propagation before either system
   script is recommended for use.
4. Make update use safe sync rather than normal bootstrap.

### Phase 1 — Unify lifecycle state

1. Introduce one declarative source/destination/profile/platform manifest.
2. Make the stored profile authoritative for sync, status, hooks, and watch.
3. Scope cleanup and validation to manifest ownership.
4. Correct health and CLI exit codes.

### Phase 2 — Restore cross-platform guarantees

1. Install and enforce Neovim 0.11+.
2. Separate package profiles or document them as independent from link profiles.
3. Fix fresh Zsh cache/runtime paths and derive the repository location.
4. Remove machine-specific Kitty and shell paths.

### Phase 3 — Make Neovim reproducible

1. Commit `lazy-lock.json`.
2. Fix Neotest, Docker, native LSP hooks, and provider-path drift.
3. Reformat the Lua tree and enforce StyLua.
4. Add a headless Neovim smoke test.

### Phase 4 — Tighten maintenance quality

1. Reduce the global Git ignore and provision Git LFS consistently.
2. Extend linting/tests to helper scripts.
3. Rewrite README setup/update/profile claims around the corrected behavior.
4. Archive or prune resolved entries from the historical backlog.
